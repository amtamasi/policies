Bookmarked — Privacy Policy

Effective July 10, 2026

What Bookmarked is

Bookmarked is an iOS app for tracking the books you read: a private shelf (To Be Read / Reading / Read), follows on books, authors, and series so you get release alerts, a yearly reading goal, CSV imports from Goodreads or StoryGraph, and requests for books missing from the catalog. Bookmarked needs an account, because your shelf lives on a server I run — that's what lets it survive phone changes and send alerts. This page describes exactly what that server and the app store, and who else touches it.

Data we collect

Account. Your email address and a user ID. Optionally, a display name and a reading-goal target and year — you type those in and can change them anytime. Your password is held by Supabase, the sign-in service; my server passes it through when you sign in and never stores it.

Your content — all of it created by you, in the app:

  • Shelf entries: which book, its status (To Be Read / Reading / Read), and started/finished dates.
  • Follows: the books, authors, and series you follow, and your alert settings for each (day-before alert, release-day alert, paused).
  • Catalog requests: the title, author, series, and any notes you type when requesting a book the catalog doesn't have, plus the request's status.
  • CSV imports: the file you pick is parsed and turned into shelf entries; titles that don't match the catalog are filed as catalog requests. The file itself is not stored — the server log records only its name and size.
  • Zero-result searches: if a catalog search finds nothing, the query text (up to 120 characters) is saved with your account so I know which books to add. Searches that return results are not stored.
  • Push registration, only if you enable notifications: a push subscription ID, your platform (iOS or Android), and your device's timezone, so alerts arrive at a sensible hour.

Automatic. Every stored record carries created/updated timestamps, and the server keeps a record of each release alert it queued for you and whether it was delivered. The API writes request logs (method, path, status, timing). Passwords and tokens are scrubbed from those logs before they are written, but sign-in and sign-up attempts do log the email used — and, like any web server, mine sees your IP address.

Where your data lives

The database is managed Postgres hosted by PlanetScale; per its connection configuration it runs in a US region (AWS us-east-2). The API runs in a container on an Oracle Cloud Infrastructure server I operate, behind HTTPS. Your sign-in credentials live with Supabase. On your phone, the app keeps your session tokens and a few preferences (theme, which onboarding hints you've seen) in its private local storage; your shelf itself is only cached there while the app runs.

Third-party services

  • Supabase — sign-in. Holds your email and password, issues the session tokens the app uses, and sends the password-reset email when you request one.
  • PlanetScale — hosts the database, so everything listed above except your password is stored on their infrastructure.
  • Oracle Cloud Infrastructure — hosts the API server and its logs.
  • OneSignal — push notifications. The OneSignal SDK runs inside the app and is given your Bookmarked user ID when you sign in. If you turn notifications on, it creates a push subscription for your device, and my server sends each alert's content (book title, release info) to OneSignal for delivery.
  • Sentry — crash and error reporting, in both the app and the API. When something breaks, Sentry receives the technical details; server-side errors are tagged with the signed-in user's ID so I can tell which account hit a bug, with the SDK's include-personal-data option switched off. In the app, Sentry is configured to capture a short screen replay only when an error actually occurs — never routine sessions.
  • Google Books and Hardcover — where the book catalog comes from. I run ingestion scripts on the server that pull titles, authors, series, release dates, descriptions, and cover-image URLs from these sources. The app never sends your searches, your shelf, or anything else about you to them, with one ordinary exception: cover images load directly from the image servers those catalog records point at, so displaying a cover is a normal image request from your device — that host sees your IP address, like any image on the web.

That is the full list. There are no analytics SDKs, no ad SDKs, and no payment processing in the app or the API. The API has an email-digest feature, but in this version no email provider is connected to it — it writes files to a server-side outbox and nothing is sent. The only email Bookmarked actually sends you is Supabase's password reset.

How your data is used

To run the app: sign you in, show your shelf, deliver the alerts you asked for, and process your imports and requests. I read catalog requests and zero-result searches — including counts grouped across users — to decide which books to add to the catalog, and I look at simple aggregate counts (like most-followed authors) computed from my own database. Crash reports are used to fix bugs. That's it. Your data is not sold, not used for advertising, not shared with data brokers, and there is no cross-app tracking.

Deleting your account and data

In the app: Profile → Delete account, then confirm. The server first deletes your Supabase sign-in identity, then your user record, and the database removes everything tied to it in the same step — shelf, follows, registered devices, catalog requests, saved searches, and queued alerts. The app signs you out and clears its local session. If you'd rather not use the button, or can't, email me at the address below and I'll delete the account by hand.

Data retention

Your data is kept for as long as your account exists and deleted when you delete the account — nothing in the code expires or archives it on a schedule. Two honest caveats: server request logs (which can include the email used on sign-in attempts and your IP address) persist on the server until rotated as part of normal operations, and the providers named above keep their own operational backups on their own schedules, which I don't control precisely.

Children

Bookmarked is not directed at children under 13, and I don't knowingly collect their data.
